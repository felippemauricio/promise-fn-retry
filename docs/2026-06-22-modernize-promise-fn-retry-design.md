# Design — Modernising `promise-fn-retry`

**Date:** 2026-06-22
**Status:** Approved

## Goal

Bring `promise-fn-retry` up to the 2026 ecosystem: modernise the language,
build, tests, lint, CI/CD, types and documentation; add new **opt-in**
features; and keep **broad compatibility** (old and current Node, browser and
backend) **without breaking existing users**.

## Current state

- Source: `src/index.js` (~80 lines, plain JS) — a single `retry(promiseFn, options)`
  function with exponential backoff (each retry doubles the delay).
- Build: Babel (`@babel/preset-env` + `add-module-exports`) → CommonJS in `lib/`.
- Tests: Jest 27 (10 tests).
- Lint: ESLint 7 + `airbnb-base` + `babel-eslint`.
- CI: Travis (Node 11.9.0) + Coveralls.
- Types: none bundled; TypeScript consumers depend on the external
  `@types/promise-fn-retry` package.
- Published on npm: v1.0.11.

## Compatibility (front + back, old + current Node)

- The source uses universal APIs only: `Promise`, `setTimeout`/`clearTimeout`
  and (optionally, when the caller passes one) `AbortSignal`. No Node-only API.
- We author modern TypeScript (`?.`, `??`); **tsup/esbuild transpiles to
  ES2015**, so it runs in old browsers and Node ≥ 12.
- `engines.node` = `>=12`. The CI matrix runs the Node lines the dev toolchain
  supports (20/22 — Vitest 4 needs Node 20+); the build target guarantees
  support for older runtimes down to Node 12.
- No runtime dependencies (zero-dep) and no `@types/node` dependency in the
  public types — keeping the library neutral between browser and backend.

## Library architecture

A single, recursive, stateless function:

```
retry<T>(fn: () => Promise<T>, options?: Options): Promise<T>
```

- `fn` is called; on rejection, a retry is decided based on `times`,
  `shouldRetry` and `signal`. `onRetry` runs on each retry. The delay grows
  exponentially (base × `backoffFactor`), with an optional cap (`maxDelayTime`)
  and optional jitter applied **at wait time**.
- Pure helpers: `buildOptionsParsed`, `buildOptionsToRetry`, `calcNewDelay`,
  `shouldRetryByExecutedTimes`, `computeWaitTime`, `delay`.

### Preserved behaviour (no breaking change)

With the defaults, behaviour is **identical to v1**: `times=1`,
`initialDelayTime=100`, backoff factor `2`, no cap, no jitter, no signal.
Existing users notice no difference.

## New features (all opt-in, additive)

| Option          | Type          | Default     | Behaviour                                                                                                                                                 |
| --------------- | ------------- | ----------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `backoffFactor` | `number`      | `2`         | Multiplier applied to the delay on each retry. `2` = doubles (current); `1` = linear; `3` = aggressive.                                                   |
| `maxDelayTime`  | `number`      | `Infinity`  | Cap on the delay between attempts (ms). Avoids huge waits under exponential backoff.                                                                      |
| `jitter`        | `boolean`     | `false`     | Randomises the delay (_equal jitter_: `wait/2 + random(0, wait/2)`) to avoid a _thundering herd_.                                                         |
| `signal`        | `AbortSignal` | `undefined` | Cancels the retries. If already aborted, rejects without calling `fn`; if aborted during a wait, rejects and clears the timer. Works in browser and Node. |

Effective-delay computation order: exponential base → `min(base, maxDelayTime)`
→ apply jitter (when enabled). The exponential base is kept "pure" (no jitter
accumulated) between attempts.

### Public types (new, bundled)

```ts
export type OnRetry = (error: unknown, options: ResolvedOptions) => void;
export type ShouldRetry = (error: unknown) => boolean;

export interface Options {
  times?: number; // default 1
  initialDelayTime?: number; // default 100 (ms)
  backoffFactor?: number; // default 2
  maxDelayTime?: number; // default Infinity
  jitter?: boolean; // default false
  signal?: AbortSignal; // default undefined
  onRetry?: OnRetry | null;
  shouldRetry?: ShouldRetry | null;
}

export type ResolvedOptions = Required<Omit<Options, 'signal'>> & {
  signal?: AbortSignal;
  retained: number;
  currentDelay: number | null;
};

export default function retry<T>(fn: () => Promise<T>, options?: Options): Promise<T>;
export { retry };
```

## Packaging

- tsup generates `dist/index.js` (ESM), `dist/index.cjs` (CJS) and the
  declaration files (`dist/index.d.ts`, `dist/index.d.cts`), ES2015 target.
- `package.json`: `type`, `exports` map (per-condition `import`/`require` with
  their own `types`), `main`, `module`, `files: ["dist"]`,
  `engines: { node: ">=12" }`, `sideEffects: false`, `packageManager`.
- Interop: `require('promise-fn-retry')` still returns the function (v1 compat),
  via a CJS footer that reassigns `module.exports = retry`; `import retry` and
  `import { retry }` both work.

## Tests

- The 10 existing tests migrated to Vitest (close to 1:1), hardening `try/catch`
  cases into `await expect(...).rejects`.
- New tests for the features, capturing delays with
  `vi.spyOn(globalThis, 'setTimeout')` (deterministic):
  - `backoffFactor` → correct delay progression.
  - `maxDelayTime` → delays bounded by the cap.
  - `jitter` → each delay within `[base/2, base]`.
  - `signal` → already-aborted does not call `fn`; aborting during a wait
    rejects and stops.
- v8 coverage with high thresholds.
